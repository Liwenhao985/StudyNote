**0. 先准备（一次性，30分钟）**

1. 进入项目目录：cd E:\nanoclaw\nanoclaw-main。
2. 新建学习目录：mkdir learning。
3. 先只建3个文件：learning\启动记录.md、learning\系统地图.md、learning\链路走读.md。
4. 固定学习节奏：每次90分钟，每周4-6次。
5. 固定止损：某阶段超时50%就缩范围，不硬顶。

**1. 每次90分钟固定流程（每次都一样）**

1. 10分钟：回顾上次记录（已确认/不确定/待验证）。
2. 65分钟：做当次主任务（只做一件事）。
3. 10分钟：记录证据（命令、日志位置、代码位置）。
4. 5分钟：写下次开始第一步（避免下次卡住）。

------

**2. 阶段A：先读规则和约束（2-4小时）**

1. 按顺序读这4个文件：
   [AGENTS.md](app://-/index.html?hostId=local)
   [README.md](app://-/index.html?hostId=local)
   [CONTRIBUTING.md](app://-/index.html?hostId=local)
   [REQUIREMENTS.md](app://-/index.html?hostId=local)
2. 在 learning\启动记录.md 建3段：必须遵守、禁止操作、常用命令。
3. 写出你自己的“红线清单”5条（例如不能破坏性改动、先构建再运行等）。
4. 完成标准：你能不看文档说出“哪些能做、哪些不能做、出问题先查哪里”。

------

**3. 阶段B：环境跑通（6-12小时）**

1. 执行安装：npm install。
2. 执行构建：npm run build。
3. 执行运行：npm run dev。
4. 每次报错都按模板记到 learning\启动记录.md：现象 -> 原因 -> 解决命令 -> 是否复现。
5. 跑通一次最小流程（能看到服务正常启动+有一次请求处理日志）。
6. 完成标准：你能在新终端从零独立再跑通一遍。

------

**4. 阶段C：系统全景图（5-8小时）**

1. 只看核心文件：
   [index.ts](app://-/index.html?hostId=local)
   [router.ts](app://-/index.html?hostId=local)
   [ipc.ts](app://-/index.html?hostId=local)
   [task-scheduler.ts](app://-/index.html?hostId=local)
   [db.ts](app://-/index.html?hostId=local)
   [container-runner.ts](app://-/index.html?hostId=local)
   [registry.ts](app://-/index.html?hostId=local)
2. 在 learning\系统地图.md 写三块：输入层、处理层、存储层。
3. 每个模块写1-3句：职责、输入、输出。
4. 补一张“消息流转顺序”：收到消息 -> 路由 -> 调度/容器 -> 回复。
5. 完成标准：你能讲清“消息从哪里进，在哪里处理，最后怎么出去”。

------

**5. 阶段D：术语和边界（3-5小时）**

1. 新建 learning\术语边界.md。
2. 先写10-20个术语：术语名 | 定义 | 代码位置。
3. 写边界清单：空输入、超长输入、重复触发、超时、外部失败、权限异常。
4. 每个边界写“系统预期行为”和“怎么验证”。
5. 完成标准：随机抽5个术语，你能定义+定位代码位置。

------

**6. 阶段E：选主链路和备链路（2-3小时）**

1. 列出3-5条候选链路（例如消息处理、任务调度、容器调用）。
2. 每条按1-5分评分：业务价值、调用频率、故障影响、可观测性、改动可行性。
3. 选1条主链路+1条备链路（要求可观测性和可行性都>=3）。
4. 把评分表写进 learning\链路走读.md。
5. 完成标准：你能解释“为什么选这两条”。

------

**7. 阶段F：主/备链路走读（6-10小时）**

1. 按固定顺序走读主链路：入口 -> 路由/调度 -> 服务 -> 数据层 -> 外部集成。
2. 打断点跑通一次，记录关键变量和关键日志位置到 learning\链路走读.md。
3. 用同样方法走读备链路。
4. 写“主备差异表”：相同点、不同点、风险点、排查优先级。
5. 完成标准：你能不看代码口述两条链路并再次跑通主链路。

------

**8. 阶段G：改动闭环（10-16小时）**

1. 目标固定：1个真实需求 + 1个小修复（保底）。
2. 每次改动前先写4项：目标、验收标准、影响范围、回滚步骤。
3. 改完后做自测并记录证据（命令、日志位置、结果）。
4. 条件检查（按适用触发）：
   仅涉及写操作才查幂等/事务。
   仅涉及权限逻辑才查鉴权/越权。
   仅涉及用户数据才查脱敏。
5. 做一次“伪评审”：自己写一段“这次改动最大风险是什么”。
6. 完成标准：两次改动都能完整复盘“为什么改、怎么测、失败怎么退”。

------

**9. 阶段H：上线模拟和复盘（4-6小时）**

1. 新建 learning\上线模拟与复盘.md。
2. 写上线前清单：冒烟、回归范围、监控点、回滚步骤、已知风险。
3. 没监控平台就用本地替代：关键日志完整性、错误次数、主流程耗时。
4. 做一次“模拟发布演练”（按清单逐项打勾）。
5. 写复盘三段：最有效方法、最大卡点、下次怎么提前规避。
6. 完成标准：你能独立完成一个小需求的“可上线模拟”闭环。

------

**10. 你每天只要填这个模板（10分钟）**

```
今天已确认： - （事实）+（证据位置） 今天不确定： - （猜测） 明天待验证： - （要验证什么）+（怎么验证） 是否触发止损： - 否/是（如果是，缩了什么范围）
```
